# 音声入力設計

## 目的

ゲーム内の既存テキストコマンド入力に、ローカル音声認識による別入口を追加する。音声認識結果は最終的に文字列として `Game.eval_text()` へ渡し、購入、使用、選択待ち、移動、戦闘などの既存 NLP 処理を再利用する。

最初の実装範囲は push-to-talk と最小限の音声状態 UI のみとする。常時録音、自動発話区切り、音声コマンド履歴の詳細表示は初回実装には含めない。

## 採用方針

第一候補は `faster-whisper` を使うローカル認識とする。短い日本語ゲームコマンドを対象にし、`language="ja"`、`condition_on_previous_text=False`、`vad_filter=True` を基本設定にする。

録音は `sounddevice` で行う。pygame のフレームループを止めないため、録音結果の認識は worker thread へ逃がし、ゲーム本体には認識済みテキストだけを queue 経由で渡す。

## コンポーネント

### `VoiceInput`

音声入力の状態、録音バッファ、認識 worker、結果 queue を持つ。

主な状態:

- `idle`: 録音していない。
- `recording`: push-to-talk キー押下中、マイク入力をバッファへ積む。
- `transcribing`: 録音終了後、worker thread が音声認識中。
- `error`: マイク、依存ライブラリ、モデル読み込み、認識処理のいずれかが失敗した。

主な責務:

- 録音開始と終了を明示的に受け取る。
- 音声バッファの所有権を録音側から認識 worker へ移す。
- 認識結果を `recognized_text_queue` に積む。
- UI 表示用に現在状態、最後の認識テキスト、最後のエラーメッセージを読み取り専用で公開する。
- エラーは例外を外へ漏らさず、短いエラーメッセージとして queue または状態へ保持する。

### `Game`

pygame event で push-to-talk キーを処理する。初回候補は `V` キー。

主な責務:

- `KEYDOWN V`: `VoiceInput.start_recording()` を呼ぶ。
- `KEYUP V`: `VoiceInput.stop_recording_and_transcribe()` を呼ぶ。
- 毎フレーム `VoiceInput.poll_result()` を呼び、認識テキストがあれば `eval_text(text)` へ渡す。
- 毎フレーム `render_voice_input_status()` を呼び、音声入力状態を画面下部の入力欄付近へ描画する。
- 音声認識中も通常の描画、移動、戦闘を止めない。

### `TextUtils`

音声認識結果にも既存の文字処理境界を適用する。英数字の幅変換は `normalize_ascii_width()`、NLP 向けの正規化は既存の `normalize_text()` に任せる。

## 音声入力 UI

音声入力 UI は、ゲーム画面下部のテキスト入力欄付近に 1 行のステータスとして表示する。初回実装では描画座標を `(300, 510)` とし、テキスト入力欄、アイテム欄、NLP 分類結果、行動結果とは別の表示領域に置く。

表示文言:

- `idle`: `Vキーで音声入力`
- `recording`: `録音しています`
- `transcribing`: `音声を認識しています`
- 認識成功直後: `認識: <recognized_text>`
- エラー: `音声入力エラー: <short_message>`

`recognized_text` と `short_message` は 40 文字を上限に表示し、超えた部分は末尾を省略する。完全な内容が必要な場合は console に出す。

UI は状態を表示するだけで、ゲーム状態を変更しない。認識成功時のゲーム状態変更は、従来どおり `eval_text()` が担当する。

## データフロー

1. プレイヤーが `V` を押す。
2. `Game` が `VoiceInput` を `recording` にし、UI に `録音しています` を表示する。
3. `sounddevice` の入力 callback が PCM データを録音バッファへ積む。
4. プレイヤーが `V` を離す。
5. `VoiceInput` が録音バッファを固定し、worker thread に渡し、UI に `音声を認識しています` を表示する。
6. worker thread が `faster-whisper` で日本語文字列へ変換する。
7. 認識結果を queue に積む。
8. `Game` が queue を読み、UI に `認識: <recognized_text>` を表示し、`eval_text(recognized_text)` を呼ぶ。

## エラー処理

マイク権限がない、入力デバイスがない、依存ライブラリがない、モデルがない、認識に失敗した、録音が短すぎる、無音だった、というケースを区別する。

ゲームループは止めない。エラーは音声入力 UI に短く表示し、詳細ログは console に出し、音声入力状態を `idle` に戻す。認識失敗時は `eval_text()` を呼ばない。

## テスト方針

実マイクと実モデルはユニットテストに含めない。テストでは fake recorder と fake recognizer を使い、次の境界を固定する。

- `KEYDOWN V` で録音開始が呼ばれる。
- `KEYUP V` で録音終了と認識開始が呼ばれる。
- 認識済みテキストが queue にある場合、`Game` が `eval_text(text)` を呼ぶ。
- `recording` では `録音しています`、`transcribing` では `音声を認識しています` が描画対象になる。
- 認識エラーでは `eval_text()` を呼ばず、表示用エラーだけを設定する。
- 既存のテキスト入力と選択待ち入力は壊さない。

手動確認では、短いコマンドを対象にする。

- 「スライムへ移動」
- 「ゴブリンを攻撃」
- 「ポーションを買う」
- 選択待ち後に「2 2」または「MP 2」

## リスク

`faster-whisper` は初回モデルロードが重い。起動時ロードにすると起動が遅くなり、初回音声入力時ロードにすると初回だけ待ち時間が出る。初回実装では遅延ロードを採用し、認識中表示で待ち時間を明示する。

`sounddevice` は OS のマイク権限と入力デバイス設定に依存する。エラー文はゲーム内表示に留め、詳細ログは console に出す。

音声認識は誤認識する。既存の曖昧ポーション選択と同じく、認識結果が対象を一意に決められない場合は既存の質問フローに入る。

## 初回実装の完了条件

- `V` を押して話し、離すと認識テキストが既存の `eval_text()` に流れる。
- 録音中は `録音しています`、認識中は `音声を認識しています` が表示される。
- 認識成功後は認識テキストが短く表示され、認識失敗時は音声入力エラーが表示される。
- 認識中に pygame の描画とゲーム更新が止まらない。
- マイクまたは認識モデルが使えない場合でもゲームが落ちない。
- fake recognizer による回帰テストが通る。
